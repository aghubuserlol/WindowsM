# WindowsM

Windows 11 ARM64 installer for Apple Silicon Macs. A SwiftUI wizard that
partitions an external SSD, deploys Windows onto it, installs the m1n1 +
U-Boot boot chain, and walks you through the parts Apple makes you do by
hand.

> Experimental. The pipeline runs end to end, but bare-metal Windows on
> Apple Silicon also depends on community firmware and drivers that are
> still moving targets. Expect to get your hands dirty. Everything
> destructive targets external disks only, the helper refuses internal
> disks outright.

## How it boots

iBoot doesn't boot loose EFI binaries off a FAT partition like a PC would.
It only starts boot objects registered in a personalized APFS volume
group's LocalPolicy. The workaround (same one the Asahi Linux installer
uses): install a throwaway macOS onto a small partition of the external
disk, then replace its kernel with m1n1 using `kmutil configure-boot`.
The internal disk is never written to. Unplug the SSD and it's all gone.

```
iBoot -> m1n1 stage 1 (in the "WinM Stub" install's boot slot)
          -> chainload -> ESP m1n1/boot.bin (stage 2: m1n1 + U-Boot UEFI)
              -> EFI/Microsoft/Boot/bootmgfw.efi -> Windows 11 ARM64
```

Boot Windows: hold the power button at startup, pick "WinM Stub".
Back to macOS: hold the power button, pick Macintosh HD.

## Building

```sh
brew install xcodegen
xcodegen generate
open WindowsM.xcodeproj
```

Builds out of the box with ad-hoc signing; the wizard UI, disk discovery
and the UUP download flow work immediately. Actually installing Windows
also needs:

1. Bundled binaries. Run the scripts (see `Sources/Resources/RESOURCES.md`):
   `build-wimlib.sh` (required), `build-m1n1.sh`, `build-edk2.sh`,
   `fetch-uup-converter.sh` (download path only), `fetch-drivers.sh`.
2. Real code signing if you want the SMJobBless helper path. Set
   `DEVELOPMENT_TEAM` and `CODE_SIGN_IDENTITY` in `project.yml`, tighten the
   requirement strings in the two Info.plists, regenerate. Unsigned local
   builds use the single-admin-prompt script path instead, which works fine.

## What the installer does

| Stage | Where | How |
|---|---|---|
| Helper install | app | SMJobBless + admin prompt (signed builds) |
| ISO mount | app | `hdiutil attach -plist -readonly` |
| Partitioning | root | `diskutil partitionDisk GPT`: auto ESP s1, APFS "WinM Stub" s2, MSR s3, WINDOWS s4. `gpt` retype of the MSR, bundled `mkntfs` for s4 |
| Image deploy | root | `wimlib-imagex apply install.wim N /dev/diskXs4`. wimlib's NTFS-3G mode writes the volume through the block device, so macOS's read-only NTFS driver doesn't matter |
| EFI boot files | root | `wimlib-imagex extract /Windows/Boot/EFI` -> `EFI/Microsoft/Boot/` + `EFI/BOOT/BOOTAA64.EFI` |
| Boot chain stage 2 | root | `m1n1.bin` + `edk2-apple.fd` concatenated (m1n1 payload convention) -> `ESP/m1n1/boot.bin`, plus BCD template and drivers if bundled |
| macOS stub | you | Boot Setup step: install macOS onto "WinM Stub" (reboots once), lower its Startup Security in recoveryOS |
| Boot registration | root | `wm-register-boot.sh`: m1n1 + a chainload variable, registered with `kmutil configure-boot`. If macOS insists on recoveryOS for this, the app prints the exact command to run there |

## Things to know

- Startup Security has to be lowered by hand, in recoveryOS, with physical
  presence. That's an Apple rule, not a missing feature. It only applies to
  the stub volume; the internal disk stays at Full Security.
- The macOS stub install reboots the machine and runs through Setup
  Assistant once. That's what personalizes the volume and creates its
  LocalPolicy. The Boot Setup step is re-entrant, so relaunch the app
  afterwards and it picks up where it left off. An Asahi-style IPSW stub
  builder could cut this to ~2.5GB and no reboot, but the full installer
  route needs zero custom APFS code, so that's what ships.
- Whether `kmutil configure-boot` accepts the swap from the running OS is
  the unproven hop. Asahi does its equivalent from inside recoveryOS. If
  kmutil refuses, the app hands you a one-line command for the recoveryOS
  terminal and the flow still completes.
- Hardware support: M1 and M2 have the full working chain. M3 is mostly
  there in upstream m1n1 but lacks device trees. M4 is the bring-up
  frontier, see [experimental/t8132-bringup](experimental/t8132-bringup/README.md).
- The "EDK2" slot is actually U-Boot for now. The old AppleSiliconPkg EDK2
  port was deleted from GitHub and its Project Mu successor can't hand off
  to the Windows kernel yet, so `build-edk2.sh` builds the thing that
  actually works: U-Boot from the Asahi chain, whose UEFI implementation
  chainloads `BOOTAA64.EFI`. Swap in a real EDK2 build when one exists; the
  helper's concatenation logic doesn't change.
- BCD: a Boot Configuration Data store is a registry hive referencing
  partition GUIDs and macOS has no tooling to author one. Generate one on
  any Windows box (`bcdboot C:\Windows /s X: /f UEFI`), drop it in
  `Sources/Resources/BCD`, and the installer stages it. Without it the boot
  manager has nothing to read; the app logs a warning.
- Drivers: there is no public driver pack for Windows on bare-metal Apple
  Silicon. Whatever you put in `drivers/` gets staged on the ESP so it's
  reachable from inside Windows (`pnputil /add-driver X:\drivers\*.inf
  /subdirs /install`).

## Security model

The app never runs as root. Privileged operations go over XPC to
`com.windowsm.helper`, which exposes a fixed set of validated operations,
not a generic "run this as root". The helper checks the code signature of
every connecting process (audit token -> `SecCodeCheckValidity`) before
accepting the connection, validates BSD names against strict patterns, and
refuses internal disks unconditionally. The unsigned-build path does the
same steps in one bundled script after a single admin prompt.

## Disclaimers

This erases disks. It is experimental. Windows licensing is your problem.
GPU acceleration and most drivers don't exist publicly for Windows on
bare-metal Apple Silicon; a VM (UTM, Parallels, VMware Fusion) is still the
practical way to run Windows on these machines.
