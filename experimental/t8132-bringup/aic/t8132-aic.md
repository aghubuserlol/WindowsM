# T8132 AIC (v3), register model and parameters

The Apple Interrupt Controller on T8132 (M4) reports ADT `compatible = "aic,3"`.
m1n1 already ships an AICv3 driver (`src/aic.c`, `aic23_init(3, …)`), but it
is **fully ADT-driven**: it reads the register-layout offsets at runtime rather
than hard-coding them per SoC. This document records those offsets for T8132,
extracted from the live ADT by `extract-aic-params.py`, so the chip's
interrupt controller is fully described for a Linux `aic,3` binding and for
review of m1n1's path.

## Parameters (extracted from a Mac mini M4, Mac16,10)

| ADT property | Value | Meaning (per m1n1 `aic.c` / `aic_regs.h`) |
|---|---|---|
| `compatible` | `aic,3` | third-generation AIC |
| `reg` (arm-io offset) | `0x181000000`, size `0x1cc000` | single MMIO window -> abs `0x381000000` |
| `cap0-offset` | `0x4` | CAP0: `NR_IRQ` (bits 15:0), `LAST_DIE` (27:24) |
| `maxnumirq-offset` | `0xc` | INFO3: `MAX_IRQ` (15:0), `MAX_DIE` (27:24) |
| `aic-iack-offset` | `0x40000` | event/IACK register (read to acknowledge) |
| `extint-baseaddress` | `0x10000` | per-IRQ target-config array base (`IRQ_CFG`) |
| `extintrcfg-stride` | `0x4a00` | per-die stride of the ext-int config block |
| `intmaskset-stride` | `0x4a00` | per-die stride, mask-set |
| `intmaskclear-stride` | `0x4a00` | per-die stride, mask-clear |
| `hwintmon-stride` | `0x4a00` | per-die stride, hw interrupt monitor |
| `aicglbcfg-offset` | `0x14` | global config register |
| `rev-offset` | `0x0` | revision register |
| `#main-cpus` | `0xa` (10) | CPUs the AIC fans interrupts out to |

## Independent cross-checks (why these are trustworthy)

1. **`extint-baseaddress == 0x10000` is exactly m1n1's `AIC3_IRQ_CFG`**
   (`aic_regs.h`). The ADT value and m1n1's independently-derived constant
   agree, this is a genuine AICv3, and m1n1's driver will compute the same
   register addresses this chip uses.
2. **`#main-cpus == 10` matches the CPU topology three ways:** the ADT `cpu@`
   nodes (6 E + 4 P), the pmgr `ECPU0..5`/`PCPU0..3` power domains, and now
   the AIC fan-out count. Three unrelated ADT regions agree.
3. **Single `0x1cc000` window with iack at `0x40000`** is the AICv3 shape
   (v2 split core/event into two reg regions; v3 folds them into one window,
   which is why the ADT exposes `aic-iack-offset` instead of a second `reg`).

## What this enables / what remains

- **m1n1:** nothing to add, its AICv3 driver reads exactly these properties
  from the real ADT at boot. This document confirms the values are present and
  sane, so the AIC is not a blocker for m1n1 reaching a usable state on M4.
- **Linux:** there is no upstream `apple,aic3` binding/driver yet. The kernel's
  `irq-apple-aic.c` handles `apple,aic` and `apple,aic2`; a v3 addition is a
  bounded delta (single-window addressing + the `0x40000` iack offset). The
  values here are the data that delta needs. Writing and *compile-verifying*
  that kernel patch needs a kernel tree and is left as the next step.
- **Windows:** Windows ARM64 expects a GIC; bridging Apple's AIC to what the
  Windows HAL wants is a separate, unsolved problem and is **not** addressed
  here.
